# S09-T01: Root-Cause Memo ŌĆö Plugin Registry v1 All-404s

**Date:** 2026-04-17
**Worker:** `plugins/.workers/plugins-registry/src/index.js`
**Production URL:** `https://plugins.nself.org`
**Investigated by:** Implementer agent

---

## Summary

The Cloudflare Worker code is well-structured and functionally correct for the endpoints it
implements. The 404s in production stem from two separate problem classes: (1) the worker was
never deployed after its most recent code additions, and (2) several endpoint paths that callers
expect are unhandled in the router and fall through to the 404 catch-all response.

---

## Root Causes

### RC-1 (PRIMARY): Worker never deployed to production

**Evidence:**
- Last commit touching `src/index.js` is `4b6d51b` dated 2026-03-09 ŌĆö over five weeks before
  today (2026-04-17).
- `deploy.sh --production` runs `npx wrangler deploy --env production`, but there is no record
  of this having been executed since the manifest endpoint and GH_ACCESS_TOKEN rename were added.
- `wrangler.toml` carries the correct route `plugins.nself.org/*` and a populated KV namespace
  ID (`fddaf7f451f54f82a7d3712ae92882ea`), so infrastructure is configured correctly ŌĆö deployment
  execution is the missing step.
- **This single cause would produce 404s on every endpoint**, because Cloudflare would return its
  own default 404 for an undeployed route, or serve a stale worker version that predates the
  current routing table.

**Fix:** Run `cd plugins/.workers/plugins-registry && ./deploy.sh --production`

---

### RC-2: `GET /plugins` (no plugin name) is unhandled

**Code path (`src/index.js` line 95):**
```js
if (method === 'GET' && path.startsWith('/plugins/')) {
```

The condition requires a trailing slash after `/plugins`, so an exact `GET /plugins` request
(path normalized to `/plugins` with no trailing slash) does not match this branch and falls
through to the 404 catch-all on line 116.

**Callers expecting this:** CLI `nself plugin list` and any consumer calling the list endpoint
by the conventional REST path `/plugins`.

**Fix:** Add a route before line 95:
```js
if (method === 'GET' && path === '/plugins') {
  return await handleRegistry(url, env, ctx);
}
```

---

### RC-3: `GET /plugins/:name/tarball` returns 404

**Code path (`src/index.js` lines 337-366, `handlePlugin`):**

```js
const parts        = path.split('/').filter(Boolean);
const pluginName   = parts[1];   // "ai"
const requestedVer = parts[2] || 'latest';  // "tarball"
```

When the CLI requests `GET /plugins/ai/tarball`, `requestedVer` is set to the literal string
`"tarball"`. The version check at line 362:
```js
if (requestedVer !== 'latest' && requestedVer !== plugin.version) {
```
evaluates `"tarball" !== "latest"` (true) and `"tarball" !== "1.0.0"` (true), so it returns
a 404 version-not-found response.

**Fix:** Add dedicated handling for the tarball sub-resource before the version check in
`handlePlugin`, or add an explicit route in the main router that intercepts
`/plugins/:name/tarball` and redirects/proxies to the plugin's `downloadUrl`.

---

### RC-4: `GET /plugins/:name/signature` returns 404

Same code path as RC-3. The path segment `"signature"` is interpreted as a version string,
fails the version equality check, and returns 404.

**Fix:** Same pattern as RC-3 ŌĆö intercept the `signature` sub-resource path and serve the
plugin's signature metadata (or a signed manifest URL) before falling into the version matcher.

---

### RC-5: `GET /plugins/revocations` returns 404

**Code path:** `path.startsWith('/plugins/')` matches, so `handlePlugin` is called with
`pluginName = "revocations"`. No plugin named "revocations" exists in either registry, so
line 358 returns a 404 plugin-not-found response.

**Fix:** Add an explicit route for `/plugins/revocations` in the main router (before the
`startsWith('/plugins/')` check) that delegates to a new `handleRevocations(env, ctx)` function.
The pro registry loader already has revocation data available; this function should filter and
return the list of revoked plugin versions.
